### 一、 CI/CD 基础与错误诊断

|**概念**|**知识点**|
|---|---|
|**CI/CD Pipeline**|持续集成/持续部署流水线。由一系列自动化任务（`build`, `test`, `security`）组成，在代码推送时自动触发。|
|**`build` 任务失败**|是 Pipeline 失败最常见的原因。通常是因为编译错误、依赖缺失或我们遇到的 **超时（Timeout）**。|
|**超时 (Timeout)**|任务运行时间超过 CI/CD 系统设定的最大限制。您的问题原因在于 `Dockerfile` 中拉取庞大的 CUDA 基础镜像和安装 PyTorch 耗时过长。|
|**解决方法**|**修改 `Dockerfile`**，使用加速镜像源 (`docker.1ms.run/...`)，或增加 CI/CD 任务的超时限制。|
|**触发机制**|CI/CD Pipeline 通常由 `git push` 事件触发，而不是手动构建。|

### 二、 Git 分支同步与偏离 (Divergence)

|**错误/概念**|**解释**|**解决方案**|
|---|---|---|
|**分支偏离**|当本地和远程分支同时拥有对方没有的提交时发生。执行 `git pull` 时会提示 `fatal: 需要指定如何调和偏离的分支`。|必须使用合并或变基策略：<br><br>  <br><br>1. **`git pull --rebase` (推荐)**：使用变基，保持历史记录线性。<br><br>  <br><br>2. **`git pull --no-rebase`**：使用合并，会创建一个合并提交。|
|**`non-fast-forward`**|推送被拒绝，因为远程仓库有新提交，而您本地落后。强制推送会覆盖远程历史。|**先 `git pull` 同步**，解决冲突后再 `git push`。|
|**合并 (Merge)**|将一个分支的历史并入当前分支，会创建一个新的 **合并提交 (Merge Commit)**。历史是网状的。|`git pull` (默认行为)。|
|**变基 (Rebase)**|将本地提交“重放”到远程最新提交之上，使历史记录保持一条直线。|`git pull --rebase`。|

### 三、 Git 冲突解决与历史修改

|**场景**|**知识点**|**操作命令**|
|---|---|---|
|**撤销已推送提交**|**最安全的方式**是创建一个新的提交来抵消原有提交的更改，不重写历史。|`git revert <提交ID>`，然后 `git push`。|
|**变基冲突解决**|在 `rebase` 过程中，Git 无法自动合并更改时会暂停。您需要手动编辑冲突文件。|1. 手动解决文件中的 `<<<<<<<`, `>>>>>>>`。<br><br>  <br><br>2. `git add <冲突文件>`。<br><br>  <br><br>3. `git rebase --continue` 或 `git rebase --skip`。|
|**修改/删除冲突**|本地执行删除操作，远程执行修改操作。Git 无法自动决定取舍。|`git checkout --theirs <文件名>` (接受远程修改) 或 `git checkout --ours <文件名>` (接受本地删除)。|
|**退出变基**|如果冲突太多或历史太混乱，可以退出重来。|`git rebase --abort`。|

### 四、 Pipeline 触发疑难杂症

|**问题**|**原因分析**|**解决方案**|
|---|---|---|
|**合并提交未触发**|CI/CD 系统默认排除合并提交，以避免冗余构建和优化资源。|**进行一个非合并提交**：修改 `Dockerfile` 或其他被 CI/CD 监视的源文件，然后 `git push`。|
|**修改无关文件未触发**|CI/CD 配置了 **路径过滤 (Path Filtering)**，只监视核心代码目录的更改。|修改被 CI/CD 视为高优先级的文件，例如 `Dockerfile` 或核心 `.py` 源文件。|
